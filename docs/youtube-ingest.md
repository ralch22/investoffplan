# YouTube ingest pipeline (`scripts/youtube-ingest.mjs`)

Enumerates every catalog project that has a real, embeddable video, generates
SEO-optimized YouTube metadata, and — after a one-time OAuth handshake — uploads
pre-downloaded video files to the **Invest off Plan** brand channel and files
them into a playlist.

Node ESM, no external deps (Node built-in `fetch` + raw REST). The media
predicate mirrors `src/lib/media.ts` (`isEmbeddableVideo` / `parseMedia`).

## What the pipeline produces

For each project with an embeddable `videoUrl` it produces a YouTube snippet:

- **title** (≤ 100 chars): `"{project} by {developer} — Off-Plan in {area}, {city} | Property Tour"`
  — the `| Property Tour` suffix is dropped first, then the whole title is
  word-boundary-truncated, to stay ≤ 100.
- **description**: one-line blurb + `From AED {min}` + `Handover {q}` +
  `Details & enquiries: https://investoffplan.com/projects/{slug}` + brand line +
  relevant hashtags.
- **tags**: project, developer, area, city, `Dubai off plan`, `UAE property`,
  etc., capped at YouTube's ~500-char total budget.
- **categoryId** `19` (Travel & Events), **privacyStatus** `unlisted`,
  `embeddable: true`, `selfDeclaredMadeForKids: false`.

## Authorization note (read before uploading)

Rami is a legally authorized agent for these developer/project promo videos and
has consent to ingest them into the Invest off Plan channel. The videos are
promotional material intended for distribution. Uploading still requires the
**video file** — the pipeline does not scrape or re-host third-party YouTube
videos. Sourcing each file (downloading the self-hosted `.mp4`, or obtaining the
original file for a developer's YouTube clip) is a **separate, explicit,
Rami-consented step** performed outside this script. The uploader only touches
files you have placed in a `--from <dir>` you control, named `<slug>.<ext>`.

## Quota math

Default YouTube Data API daily quota is **10,000 units**.

| Call | Units |
|---|---|
| `videos.insert` (per upload) | 1,600 |
| `playlistItems.insert` (per video) | 50 |
| **Per ingested video** | **1,650** |

→ ~6 uploads/day before `403 quotaExceeded`. `--upload` defaults to `--limit 3`
per run and warns about the cap. Uploading all 39 pending videos ≈ 64,350 units
≈ **7 days** at the safe rate (or request a quota bump in GCP).

## One-time setup (Rami)

1. **Store the Desktop OAuth client credentials** (secure-keys handshake — the
   values never touch chat). In GCP project `corded-pivot-502106-u8`, open the
   OAuth client `investoffplan-youtube` (Desktop app), copy its **Client ID** and
   **Client secret**, and write:

   ```
   # ~/.iop-youtube-oauth   (chmod 600)
   CLIENT_ID=xxxxxxxx.apps.googleusercontent.com
   CLIENT_SECRET=xxxxxxxx
   ```

   ```bash
   chmod 600 ~/.iop-youtube-oauth
   ```

2. **Authorize** (loopback flow — opens a consent URL, catches the redirect on
   `127.0.0.1`, exchanges the code, stores the refresh token `chmod 600`):

   ```bash
   node scripts/youtube-ingest.mjs --auth
   ```

   Sign in as the **channel owner** of Invest off Plan and approve. If creds are
   missing the script prints these exact steps and exits 0. The refresh-token
   value is never printed.

## Day-to-day flow

```bash
# 1. Enumerate + (re)generate the plan. No network, no auth. Default command.
node scripts/youtube-ingest.mjs --dry-run

# 2. (Rami, separate consented step) download the source videos into a dir,
#    named <slug>.<ext>, e.g.:
#      ~/iop-videos/damac-altitude-tower.mp4
#      ~/iop-videos/1wood-residence-phase-2-by-object-1.mp4

# 3. Upload up to N pending items (default 3) and file them into the playlist.
node scripts/youtube-ingest.mjs --upload --from ~/iop-videos --limit 3
```

- `--dry-run` rewrites `data/youtube-ingest-manifest.json` (an **array** of
  `{slug, snippet, videoSource, status, videoId, …}`) and prints a summary
  (count, sample titles, estimated quota). It **preserves** the
  `uploaded`/`videoId` state of any already-ingested item — regenerating the
  snippet never clobbers upload progress.
- `--upload` is **idempotent**: each successful upload writes `status:"uploaded"`
  + `videoId` back to the manifest immediately, so re-runs skip done items and
  continue from where they left off. Slugs with no matching file in `--from` are
  skipped with a logged note (not an error).
- The script refuses ambiguous/destructive states with clear messages (missing
  creds, missing refresh token, missing manifest, missing `--from` dir) and
  **never prints secrets or token values**.

## Later: point site embeds at the ingested channel videos

Once videos live on the brand channel, the site can switch each project's
`videoUrl` from the developer source to the brand-channel watch URL
(`https://www.youtube.com/watch?v={videoId}` — captured in the manifest). Because
`src/lib/media.ts` already classifies YouTube URLs into `youtube-nocookie`
embeds, the PDP media player, the `VideoObject` JSON-LD, and the video sitemap
all keep working unchanged — only the underlying id changes. A follow-up script
could read `data/youtube-ingest-manifest.json` and write each `videoId` back into
`data/catalog.json` / `data/project-enrichments.json`, consolidating watch time,
SEO signal, and channel subscribers on the owned channel instead of the
developers'.
